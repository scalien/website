Overview

ScalienDB takes some of the best ideas of distributed systems and combines them with the dependability of SQL databases.

Distributed Database

ScalienDB uses Paxos to replicate data. See the Paxos page for more.

Next to replication, the other crucial ingredient of distributed databases is Scalability. ScalienDB uses Horizontal Scalability. This means you can add commodity servers to:

  • Increase storage space
  • Increase write throughput
  • Increase read throughput

ScalienDB supports all three scalability criteria by the addition of any number of Shard Servers on the fly. Data is split into shards and then distributed between Shard Servers transparently, maintaining the appearance of a single database to the outside.

Controllers

The Controllers act as the brain of the ScalienDB cluster and communicate among themselves to ensure the fault-tolerance and high-availability of the entire distributed database system. Shard Servers and their membership in Replication Sets are managed by the Controllers. Internally, the controller nodes use the Paxos protocol to guarantee that the ScalienDB cluster is always in a well-defined, consistent state without the possibility of a split-brain scenario. The controllers handle the following functionality:

  • Store the database schema (databases, tables)
  • Manage membership of shard servers
  • Assign Shard servers into Replication Sets for replication
  • Migrate shards across Replication Sets

You can use one, three, five or more Controllers in the ScalienDB setup depending on the level of fault-tolerance desired. We recommend to use one Controller in test environments and five Controllers in production.

Clients

The client library is smart client. It is easy for you to quickly write ScalienDB applications without worrying about the distributed nature of the database. The client library is fully aware of the state of the ScalienDB cluster and hides any failover events from you. The client supports distributing reads inside Replica Sets for linear read scalability and distributed writes across different Replicata Sets for linear write scalability. Additionally, you can batch commands for maximum throughput. ScalienDB client libraries are available for Java, C# and Python. Each client library adheres to the idioms of its language.

Data Model

ScalienDB is a NoSQL database with a key-value data model. At the highest level, data is organized into databases, then into tables, just like in traditional SQL databases. Tables are namespaces for key-value data, which may be split into several shards. Shards are mapped into Replica Sets and stored on Shard Servers. Keys and values in ScalienDB are not strongly typed, which allows for maximum flexibility when developing applications. Keys are limited to 64K length, while values may be practically unlimited in length, which means ScalienDB can be used for a wide variety of data management tasks. This architecture combines the best of relational and NoSQL models: it features databases and tables to organize data while employing a simple, flexible key-value model.

Shard servers

The Shard Servers are the workhorses of the distributed ScalienDB architecture. Shard servers are assigned into Replica Sets by Controllers for replication. Like Controllers, Paxos is employed to ensure that the data is always up to date. A high-performance linear on-disk storage engine handcrafted for ScalienDB is used to store data on disk. This storage engine is the best of both worlds. It combines technologies from traditional architectures, such as transactional redo logs to eliminate any possibility of data loss, and cutting-edge technologies such as linearized disk I/O, introduced by Google's Bigtable system. This combination of technologies assures that ScalienDB delivers both the dependability of SQL databases while opening the door to unlimited data and throughput scaling on conventional commodity hardware.

Management console

Administrative tasks are performed on the web-based Management Console using a browser like Mozilla Firefox or Google Chrome. The management console is fully featured and supports all ScalienDB operations:

  • Monitoring the cluster state
  • Monitoring catchup and shard migration progress
  • Create, rename, delete databases
  • Create, rename, delete, truncate tables
  • Create, delete Replication Sets
  • Add, remove Shard Servers from Replication Sets
  • Shard splitting and shard migration

ScalienDB has a built-in HTTP server for performing management operations. This allows you to integrate these tasks into their broader management suite using the powerful ScalienDB JSON API.
